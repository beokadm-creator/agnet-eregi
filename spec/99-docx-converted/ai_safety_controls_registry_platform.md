**AI 엔진 안전장치 스펙(상세): 등기 대행 플랫폼**

작성일: 2026-04-13

**1. 안전 목표(Top-level Safety Goals)**

1.  법률 리스크 최소화: AI는 ‘추천/초안/경고’까지만, 확정 판단은 사람 또는 규칙 엔진.

2.  개인정보/민감정보 보호: 최소수집, 마스킹/가명처리, 사건 단위 권한 통제, 감사 로그.

3.  환각/오류 전파 차단: 근거/불확실성/전제조건을 UI와 로그에 강제.

4.  프롬프트 인젝션 방어: 업로드 문서·사용자 입력을 지시가 아닌 데이터로만 처리.

5.  감사 가능성: 입력/출력/모델/정책 버전 재현 가능.

**2. 권한 모델(Authority Matrix)**

|                |                                                        |
| -------------- | ------------------------------------------------------ |
| **레벨**         | **허용/금지/승인**                                           |
| L1 안내/요약(저위험)  | 자동 노출 가능. 상태/준비물/누락/다음 단계 설명. 단정적 법률 판단/확정 견적 금지.      |
| L2 검토/진단(중위험)  | 누락/불일치/리스크 플래그 ‘추천’ 및 보완 메시지 ‘초안’. 사용자 노출/발송 전 사람 승인.  |
| L3 생성/자동화(고위험) | 서식/의사록/안내문 초안, 견적/ETA 범위. 제출/서명/선정/확정 금액은 항상 사람/규칙 엔진. |

**3. UI/UX 안전장치(오인 방지)**

- AI 카드/메시지에 ‘AI 검토(초안)’, ‘확정 아님’, ‘담당자 검토 필요’ 라벨을 고정 노출.
- 금액/시간은 단일 숫자 대신 범위 + 전제조건(‘서류 검토 완료 후 확정’).
- 근거 표시: 사건팩 규칙/업로드 문서(버전)/통계(표본) 중 무엇을 근거로 했는지 노출.
- 불확실성 표시: confidence(높음/중간/낮음) + 낮은 이유(정보 누락/스캔 품질 등).
- 정정 UX: 사용자가 ‘틀림’ 표시 시 해당 항목을 잠그고 수동검토로 라우팅.

**4. 개인정보/문서 보안(PII)**

**4.1 최소수집/목적제한**

- 사건팩이 요구하는 필드/서류 외 입력을 기본 차단(필드 레벨 강제).
- 선택 입력은 기본 숨김(필요 시만 확장).

**4.2 마스킹/가명처리**

- 자동 마스킹 대상: 주민번호/여권번호/계좌번호/전화/이메일/상세주소 등.
- 원본 보관이 필요하면 ‘암호화 원본 저장소’와 ‘AI 처리용 가명본’을 분리.

**4.3 접근통제/감사**

- RBAC(고객/수행자/운영자/관리자) + 사건 단위 권한(배정된 수행자만 접근).
- 다운로드/공유 링크 만료(시간 제한) + 다운로드 로그(누가/언제/무엇).

**5. 프롬프트 인젝션 방어(필수 패턴)**

1. 원칙: 문서/사용자 입력은 Instruction이 아니라 Evidence/Data.
2. 구조화 추출 → 판단 분리: (1) 필요한 필드만 추출(JSON) (2) 사건팩 규칙 대조로 경고/누락 생성.
3. 문서 분석 모델은 외부 도구 호출 권한 없음(웹/DB쓰기 차단).
4. 시스템 프롬프트/정책은 버전 관리 + 변경 승인 프로세스 적용.

**6. 품질/정확도(환각 방지)**

- Rule-first: 사건팩 규칙과 충돌하는 AI 출력은 자동 폐기 또는 수동검토.
- Validator: 타입 오류/필수 누락/불가능 조합 탐지.
- confidence 임계값: 낮으면 결론 대신 ‘추가 질문’만 생성.
- 고리스크 플래그는 보수적으로 노출(규칙+AI 동시 트리거 등 운영 정책 선택 가능).

**7. 감사로그/재현성(필수 저장 항목)**

- request_id, case_id, casepack_id, 문서 해시/버전
- 모델/프롬프트/정책 버전
- 입력 요약(가명 처리)과 출력(JSON)
- 사람이 수정한 내역(diff), 승인자/시각/발송 여부

**8. 사고 대응(Incident Response)**

1. 탐지: 이상 응답/클레임/유출 의심.
2. 격리: 기능 플래그 OFF, 영향 범위(사건/문서) 조회.
3. 통지: 내부/파트너/고객 커뮤니케이션 템플릿.
4. 복구: 정책/프롬프트 롤백, 데이터 정정, 테스트 추가.
5. 사후: RCA 문서화 + 재발 방지 체크리스트 업데이트.
