# 운영 최적화 체크리스트

작성일: 2026-05-03

## 1. 현재 기준선

현재 저장소와 배포 상태는 다음 기준을 만족한다.

- 브랜치 상태: `main...origin/main`
- Functions 런타임: `staging`, `prod` 모두 `nodejs22`
- 등기 퍼널 시나리오: 총 `28개` 런타임 반영 완료
- `gcf-artifacts` cleanup policy: `staging`, `prod` 모두 `1일`
- 최근 점검 기준:
  - `b2bWebhookWorker` 인덱스 문제 수정 완료
  - `ops_query_health` 인덱스 문제 수정 완료
  - 최근 `24h~72h` 기준 앱 런타임 `WARNING/ERROR` 로그는 유의미한 재발 없음

## 2. 다음 점검 타이밍

아래 시점 중 하나가 오면 운영 로그를 다시 확인한다.

- 신규 배포 직후
- 사용자 유입/상담/제출 건수가 의미 있게 늘어난 뒤
- 운영자 화면에서 응답 지연이나 빈 목록 이슈가 보일 때
- 예약 워커 실패 알림이나 수동 재처리 필요가 발생했을 때
- 최소 주 1회 정기 점검 시

## 3. 우선순위 기준

다음 순서로 대응한다.

1. 실제 사용자 요청 실패
2. 워커/배치 실패로 인한 누락 또는 재시도 적체
3. Firestore 인덱스 누락
4. 반복 경고 로그
5. 배포 도구 체인 경고

## 5. 반복 확인 포인트

### Functions

- `api`
  - 최근 사용자 요청 실패가 다시 생기는지 확인
- `b2bWebhookWorker`
  - 재시도 큐 적체나 인덱스 관련 오류 재발 여부 확인
- `documentAiStorageTrigger`
  - 실제 실행 실패가 생기는지와 버킷 연동 상태 확인
- 예약 워커 전체
  - `notifyWorker`, `opsRankingWorker`, `settlementBatchWorker`, `b2gFeeWorker` 등은 배포 실패 이력이 있었으므로 실제 실행 로그가 깨끗한지 주기적으로 확인

### Firestore

- 새로 생긴 복합 쿼리가 있으면 인덱스 누락이 발생하기 쉽다.
- `FAILED_PRECONDITION: The query requires an index`가 보이면 즉시 링크를 확보하고 `firestore.indexes.json`에 반영한다.
- `ops_query_health`에 쌓이는 항목은 새 병목의 조기 신호로 본다.

### Artifact Registry

- 배포 직후 `Unhandled error cleaning up build images` 경고가 다시 보일 수 있다.
- 하지만 `cleanup policy`가 잡혀 있으므로 실제 비용 이슈인지 먼저 패키지 목록으로 확인한다.
- 잔존 패키지가 보이면 수동 삭제 후 재확인한다.

## 6. 지금 당장 손대지 않는 항목

다음 항목은 현재 기준으로 우선순위가 낮다.

- `punycode` deprecation warning
  - 앱 코드보다 배포 도구 체인 영향이 큼
  - 최신 `firebase-tools`도 `node-fetch@2` 계열 의존을 포함
- 과거 Cloud Audit ERROR 로그
  - 현재 런타임 장애가 아니라 이전 배포 실패 이력일 수 있으므로, 최근 실행 로그와 분리해서 판단

## 7. 다음 액션 규칙

- 최근 `24h~72h` 내 재발 로그가 없는 항목은 바로 수정하지 않는다.
- 증상 없는 선제 리팩터링보다, 실제 재발 로그가 있는 함수 1건씩 잡아서 수정한다.
- 수정 후에는 아래 순서로 마무리한다.

1. 로컬 빌드/테스트
2. `staging` 선택 배포
3. 로그 재확인
4. 필요 시 `prod` 반영
5. 커밋/푸시
