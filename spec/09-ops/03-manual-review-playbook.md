# 수동검토(Manual Review) 플레이북 — v1

목표: “운영을 AI에게 맡긴다” 하더라도, 고리스크/예외 케이스는 결국 **수동검토 큐**로 모입니다.  
이 문서는 수동검토 큐에 들어온 케이스를 **일관되게 처리**하기 위한 운영 절차(결정 트리/승인/로그)를 정의합니다.

참조:
- Ops 콘솔 요구사항: `09-ops/01-ops-console-requirements.md`
- 상태 머신: `02-engine/04-state-machines.md`
- 이벤트 envelope: `02-engine/events/case_event.schema.json`
- 보안/권한: `10-security/01-auth-rbac-rls.md`

---

## 0) 원칙

1. **Rule-first**: 사건팩 규칙/정책이 우선, AI는 보조(요약/추천/초안).
2. **증빙 우선**: 판단 근거는 반드시 `domain_events`(타임라인) + 문서 버전으로 연결.
3. **되돌릴 수 있게**: 재배정/환불/정산 변경은 승인 게이트 + 롤백 플랜을 동반.
4. **PII 최소 노출**: OCR/PII는 기본 차단, 필요 시 승인 게이트로 제한 시간 접근.

## 1) 큐 진입 조건(표준)

다음 중 하나라도 해당하면 `cases.status = manual_review` 로 라우팅.

### 1.1 리스크 기반
- `RISK_FLAGGED` severity `high/critical`
- 문서 위조/불일치 신호(예: 동일 sha256 반복, 메타 불일치)
- 신원확인 필요 플래그(정책)

### 1.2 상태 충돌/정책 위반
- 파트너/사용자 간 주장 충돌(분쟁 태그)
- 광고/추천 정책 위반(허위 프로필, 부정 클릭 의심)
- SLA 반복 브리치 파트너에 대한 자동 중단 후보

### 1.3 결제/정산 이상
- 결제/환불 이벤트 불일치(대사 오류)
- 부분 환불/추가비용 분쟁

## 2) 운영 처리 흐름(표준 절차)

### Step 1) 케이스 요약 패킷 확인(3분 내)
운영 AI 또는 시스템이 생성한 “요약 패킷”을 우선 확인:
- 사건팩/현재 상태
- 마지막 10개 이벤트 타임라인
- 문서 슬롯 상태(미업로드/검토중/OK/보완필요)
- 리스크 사유(근거 링크: 이벤트/문서)
- 권장 조치 1~3개(승인 필요 여부 포함)

> 요약 패킷이 없으면, Ops는 타임라인/문서에서 직접 생성(추후 AI 자동화 목표).

### Step 2) 분류(Triage) — 케이스 타입 라벨링
아래 중 하나로 분류하고 라벨을 붙입니다(검색/통계/자동화에 필요).
- A: 문서 불일치/위조 의심
- B: 입력 정보 부족(사용자 미응답/미업로드)
- C: 파트너 품질/SLA 이슈
- D: 결제/환불/금액 분쟁
- E: 정책 위반(광고/추천/프로필)

### Step 3) 결정 트리 적용(케이스 타입별)

#### A) 문서 불일치/위조 의심
1) 문서 버전 비교(sha256/업로드 출처/시간)
2) OCR/추출 결과 열람이 필요하면 승인 게이트로 임시 권한 부여
3) 조치:
   - 사용자 보완 요청(메시지 승인 게이트)
   - 파트너 재검토 요청
   - 심각 시 케이스 동결 + 법무 검토 루트(향후)

#### B) 입력 정보 부족
1) “다음 할 일”을 사용자에게 1개만 제시(과부하 금지)
2) 조치:
   - 리마인드 발송(템플릿)
   - 일정 시간 무응답 시 케이스 종료/취소 정책 적용(사건팩 오버라이드 가능)

#### C) 파트너 품질/SLA 이슈
1) SLA 타이머 확인(브리치/임박)
2) 파트너 응답 이력/품질 지표 확인
3) 조치:
   - 파트너 경고/재배정(승인)
   - 반복 위반 시 노출 제한/정지(정책)

#### D) 결제/환불/금액 분쟁
1) 사용자 동의 로그(`quote_consents`) 확인
2) 사건 상태 머신 상 위치 확인(환불 가능 범위)
3) 조치:
   - 부분 환불 계산(정책 기반) → 승인 큐로 제출
   - 파트너 정산 차감(필요 시) → 승인 큐로 제출

#### E) 정책 위반(광고/추천/프로필)
1) 위반 유형 식별(허위 정보/부정 클릭/과장 광고)
2) 조치:
   - 광고 중단/노출 제한
   - Verified/Pro 강등/정지
   - 반복 위반 파트너 블랙리스트(향후)

### Step 4) 액션 실행(승인 게이트 포함)
운영 액션은 다음 중 하나 형태로 실행됩니다.
- L0 자동: 내부 태그/큐 이동/우선순위 조정
- L1/L2 승인: 메시지 발송/견적 확정/환불/재배정
- L3 승인: 정책/권한/모델 롤백

### Step 5) 종료/재개
처리 후 케이스를 다음 중 하나로 전환:
- `in_progress` (재개)
- `waiting_user` (사용자 대기)
- `waiting_partner` (파트너 대기)
- `cancelled` / `completed` (정책/상태에 따라)

## 3) 필수 로그/이벤트(감사)

### 3.1 Audit log(필수)
모든 수동검토 케이스는 아래 action 중 하나 이상을 남겨야 함:
- `TRIAGE_CASE`
- `VIEW_TIMELINE_JSON`
- `VIEW_PII` (해당 시)
- `REASSIGN_PARTNER`
- `APPROVE_MESSAGE_SEND`
- `APPROVE_REFUND`
- `FREEZE_CASE`
- `POLICY_ENFORCEMENT_ACTION`

필드:
- actor, reason(티켓/사유), target(caseId), meta(diff/payload hash)

### 3.2 도메인 이벤트(권장 확장)
운영상 중요한 지점은 이벤트로도 남기는 것을 권장:
- `CASE_FROZEN`, `CASE_UNFROZEN`
- `CASE_REASSIGNED`
- `PARTNER_SUSPENDED`

---

## 4) AI 자동화 포인트(운영 위임을 위한 분해)

수동검토에서 AI가 “할 수 있는 일”과 “하면 안 되는 일”:

AI 가능(자동):
- 사건 요약/타임라인 요약
- 정책 체크리스트 실행(규칙 엔진)
- 증빙 링크 수집(이벤트/문서 ref)
- 권장 액션 제안 + 위험도

AI 금지(승인 필요):
- 환불 승인/집행
- 사용자에게 나가는 메시지 발송
- 파트너 재배정/정지
- PII/OCR 원문 열람
