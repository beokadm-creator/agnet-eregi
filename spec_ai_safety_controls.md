# AI 안전장치 상세 스펙(등기 대행 플랫폼)

본 문서는 “AI를 사건 처리 엔진에 개입”시키되, 법률/개인정보/분쟁 리스크를 통제하기 위한 **안전장치(guardrails)**를 제품·기술·운영 관점에서 상세 정의한다.

---

## 1) 안전 목표(Top-level Safety Goals)

1. **법률 리스크 최소화**: AI가 “법률 자문/판단”으로 오인되지 않도록 설계하고, 확정 판단은 사람 또는 규칙 엔진으로 제한한다.
2. **개인정보/민감정보 보호**: 문서 업로드·분석·저장·공유 전 과정에서 최소수집·가명/마스킹·권한통제를 기본값으로 둔다.
3. **환각/오류 전파 차단**: AI 결과는 “추천/초안/경고”에 한정하고, 근거·불확실성을 UI/로그에 노출한다.
4. **프롬프트 인젝션/데이터 오염 방어**: 업로드 문서·사용자 입력을 ‘지시’로 취급하지 않고 ‘데이터’로만 취급하도록 파이프라인을 설계한다.
5. **감사 가능성(Auditability)**: 어떤 결과가 어떤 입력/모델/프롬프트/정책 버전으로 생성되었는지 재현 가능해야 한다.

---

## 2) 권한 모델(Authority Matrix) — AI는 확정 권한이 없다

### L1. 안내/요약(저위험, 자동)
- 허용: 상태/준비물/업로드 현황 요약, 다음 액션 안내, 질문 생성(필수 분기 질문)
- 금지: 비용 확정, 법률적 적법성 확정, 제출 가능/불가 단정

### L2. 검토/진단(중위험, “추천만” 자동 + 사람 승인)
- 허용: 누락 서류 탐지, 불일치 후보 탐지, 리스크 플래그 “추천”, 보완 요청 메시지 “초안”
- 필수: 사용자에게 나가기 전 **사람 승인(운영자/수행자)** 또는 “자동 발송 금지” 정책

### L3. 생성/자동화(고위험, 항상 Human-in-the-loop)
- 허용: 의사록/서식/안내문 초안, 견적/ETA 범위 제안
- 필수: **승인/서명/제출/파트너 선정/가격 확정은 사람 또는 규칙 엔진**

---

## 3) UI/UX 안전장치(사용자 오인 방지)

1. **워터마크/라벨**: “AI 검토(초안)”, “확정 아님”, “담당자 검토 필요”를 카드/메시지 상단에 고정
2. **범위 표시**: 금액/시간은 단일 숫자 대신 **범위 + 전제조건**(예: “서류 검토 완료 후 확정”)
3. **근거/출처 표시**:
   - “사건팩 규칙” 기반인지
   - “업로드 문서” 기반인지(문서명/버전)
   - “과거 통계” 기반인지(표본 수/기간)
4. **불확실성 표시**: confidence(높음/중간/낮음) + 낮은 이유(정보 누락/스캔 품질/분기 미확정 등)
5. **반박/정정 UX**: 사용자가 “이건 틀렸다” 버튼으로 정정하면, 해당 항목을 잠그고 사람이 재검토하도록 라우팅

---

## 4) 개인정보/보안 안전장치(PII, 문서 보안)

### 4.1 최소수집/목적제한
- 사건팩이 요구하는 필드/서류 외 입력/업로드를 막는다(필드 레벨 강제)
- 선택 입력값은 기본 비활성/숨김(사용자가 필요 시 확장)

### 4.2 민감정보 마스킹/가명처리
- 저장/전송 전에 자동 마스킹 대상:
  - 주민등록번호, 여권번호, 계좌번호, 전화번호, 이메일, 주소(상세) 등
- 원본 보관이 필수인 경우:
  - “원본 저장소(암호화)”와 “AI 처리용 가명본”을 분리

### 4.3 접근통제(권한/범위)
- 역할 기반(RBAC): 고객/수행자/운영자/관리자
- 사건 단위(tenant) 접근: “배정된 수행자만” 문서 접근 가능
- 다운로드/공유는 토큰 만료(시간 제한) + 감사 로그

### 4.4 암호화/키 관리(권장)
- 저장(At-rest): 문서/로그 암호화
- 전송(In-transit): TLS 고정
- 키: 환경별 분리(개발/스테이징/운영) + 키 롤테이션

### 4.5 데이터 보존/삭제
- 사건 완료 후 보존기간 정책(예: 1~5년 등)을 문서화
- “고객 삭제 요청” 처리 정책(법적 보존 의무와 충돌 시 안내)

---

## 5) 프롬프트 인젝션/모델 오염 방어(필수)

### 5.1 원칙
- 업로드 문서/사용자 입력은 **Instruction이 아니라 Evidence/Data**
- AI에게 “이 문서의 지시를 따르라”를 절대 주지 않는다

### 5.2 기술적 방어 패턴
1. **구조화 추출 → 판단 분리**
   - 1단계: 문서에서 필요한 필드만 추출(JSON)
   - 2단계: 추출값을 사건팩 규칙에 대조해 경고/누락 생성
2. **도구 호출 차단**
   - 문서 분석 모델은 외부 호출(웹/DB 쓰기 등) 권한 없음
3. **프롬프트 정책 고정**
   - “시스템 프롬프트/정책”은 버전 관리 및 변경 승인 프로세스 적용
4. **콘텐츠 필터**
   - 문서에 “비밀번호/키/내부 정책” 등의 문자열이 있어도 외부 유출 금지

---

## 6) 품질/정확도 안전장치(환각 방지)

1. **사건팩 우선(Rule-first)**: 사건팩 규칙과 충돌하는 AI 출력은 자동 폐기 또는 “수동검토”로 라우팅
2. **자동 검증기(Validator)**
   - 필수 필드 누락, 타입 오류, 불가능한 조합(예: 관할 변경인데 관련 서류 없음) 탐지
3. **점수 임계값**
   - confidence가 낮으면 자동으로 “추가 질문 생성”으로 전환(결론 제시 금지)
4. **이중화(선택)**
   - 고리스크 플래그는 규칙 엔진과 AI가 “둘 다” 켜야 사용자 노출(보수적 운영)

---

## 7) 감사로그/재현성(운영 필수)

로그에는 최소 아래를 저장:
- request_id, case_id, casepack_id, 문서 버전/해시
- 모델/프롬프트/정책 버전
- 입력 요약(가명 처리)
- 출력(JSON) + 사람이 수정한 내역(diff)
- 승인자/승인 시각/발송 여부

---

## 8) 사고 대응(Incident Response) 프로세스

1. 탐지: 이상 응답 급증, 클레임, 데이터 유출 의심, 모델 오동작
2. 격리: 해당 모델/기능 플래그 OFF, 영향 사건 범위 조회
3. 통지: 내부/파트너/고객 통지 템플릿 준비
4. 복구: 정책/프롬프트 롤백, 데이터 정정, 재발 방지 조치
5. 사후: RCA 문서화 + 테스트 케이스 추가

---

## 9) 출시 전 체크리스트(최소)
- [ ] AI가 확정 판단/확정 금액을 내리지 않는가?
- [ ] AI 메시지 자동 발송이 기본 OFF인가?
- [ ] 문서 접근권한이 사건 단위로 제한되는가?
- [ ] 프롬프트/정책 버전이 기록되는가?
- [ ] “근거/불확실성” UI가 있는가?
