# 구현 준비물(2): 환경/배포/비밀관리/KMS/마이그레이션/백업 — v1

목표: “개발은 되는데 운영이 안 되는” 상태를 방지하기 위해, 배포/비밀/DB 운영의 기준을 고정한다.  
원칙: 운영을 AI에게 맡기더라도, **비밀/권한/데이터 복구**는 강하게 통제되어야 한다.

---

## 0) 환경 구분

최소 3개 환경:
- `dev` : 로컬 개발
- `stage` : 스테이징(실운영 유사)
- `prod` : 운영

필수 규칙:
- prod 데이터는 stage/dev로 복제하지 않음(PII 위험). 필요 시 익명화 샘플만.
- 환경별로 다른 KMS key / DB / 스토리지 버킷 사용

---

## 1) 비밀(Secrets) 관리

### 1.1 비밀 종류(예)
- DB 접속 정보
- PG API 키/웹훅 서명 키
- 문자/카카오/이메일 발송 키
- 스토리지(S3/GCS) 키
- AI 모델 API 키/프롬프트 서명 키(선택)

### 1.2 저장 원칙(필수)
- 코드/레포에 커밋 금지
- 환경변수(.env)도 prod에서는 금지(가능하면 Secret Manager 사용)
- 로컬 개발은 `.env.local`(gitignore) 허용

### 1.3 접근 통제
- prod 비밀 조회 권한은 최소화(ops_admin 소수)
- 조회/변경은 감사 로그(누가/언제/왜)
- 키 회전(rotate) 절차 문서화(분기/반기 등)

## 2) KMS/암호화(PII/문서)

참조:
- `10-security/01-auth-rbac-rls.md`
- `10-security/02-document-storage-and-retention.md`

### 2.1 암호화 대상(필수)
- `user_pii` (전화/이메일/이름)
- 파트너 정산 계좌/세무 정보(PII)
- OCR/추출 결과(`document_ocr_results`)

### 2.2 키 계층(권장)
- Master Key(KMS): 환경별 1개(또는 테넌트별)
- Data Key(앱 레벨): 레코드/필드 단위로 파생

### 2.3 운영 요구사항
- 키 폐기/회전 시 재암호화 배치 필요(성능/다운타임 고려)
- “PII 열람 승인”은 임시 권한 토큰 + 만료

## 3) 배포(Deployment) 전략

### 3.1 배포 단위
권장:
- API 서버(오케스트레이션 포함)
- 프론트 3종(모바일/파트너/ops)
- 워커(문서/OCR/정산 배치)

### 3.2 배포 방식(권장)
- Blue/Green 또는 Rolling update
- Feature flag 기반으로 기능 점진적 공개
  - AI/결제/메시지/광고 등은 긴급 OFF 가능해야 함

참조:
- `11-platform/01-observability-and-alerting.md` (플래그/인시던트)

## 4) DB 마이그레이션(Migration) 원칙(PostgreSQL)

참조 DB 모델:
- `08-data/01-db-model.md`

### 4.1 원칙(필수)
- 마이그레이션은 **순방향 적용**이 기본(rollback은 별도 스크립트)
- prod에서는 “락이 긴 DDL” 금지(컬럼 추가는 nullable → backfill → not null)
- 인덱스 생성은 concurrent(가능 시)

### 4.2 이벤트 스토어(`domain_events`) 주의
- append-only 성질 유지(업데이트/삭제 금지)
- 보관/파티셔닝(규모 커지면 월 단위 파티션 권장)

### 4.3 RLS 정책 배포
- 마이그레이션에 RLS 정책을 포함
- 애플리케이션이 세션 변수(`current_partner_id`)를 주입하도록 표준화

## 5) 백업/복구(Backup & Restore)

### 5.1 목표(RPO/RTO)
예시(초안):
- RPO 15분(데이터 손실 허용)
- RTO 2시간(복구 목표)

### 5.2 백업 범위
필수:
- PostgreSQL 전체 백업(스냅샷 + WAL/증분)
- 오브젝트 스토리지(문서 파일) 버전/보관 정책
- 시크릿/설정(최소 메타)

### 5.3 복구 시나리오(필수 테스트)
- DB 단독 복구(특정 시점 PITR)
- 문서 스토리지 복구(특정 prefix/기간)
- 결제/정산 재구성:
  - `domain_events` + PG 웹훅 로그로 대사 가능한지 확인

### 5.4 정기 리허설(권장)
- 분기 1회 복구 리허설
- 결과를 런북으로 기록(11-platform에 연결)

---

## 6) 운영 AI와의 연결(중요)

운영 AI가 할 수 있는 것:
- 배포 후 모니터링 지표 기반 이상 탐지/요약
- 마이그레이션 전후 체크리스트 실행(읽기 전용)
- 백업/리허설 결과 리포트 생성

운영 AI가 하면 안 되는 것(사람 승인/제한):
- 비밀 조회/변경
- KMS 키 회전/폐기
- prod DB restore 실행
