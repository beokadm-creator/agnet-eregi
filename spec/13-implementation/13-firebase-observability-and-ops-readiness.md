# Firebase 구현 개발 기획(6): Observability/운영준비(로그/알림/런북) — v1

목표: Firebase 기반에서도 결제/환불/정산/승인게이트 같은 “돈/분쟁” 영역은 **운영 가능성(operability)** 이 핵심이므로, 관측/알림/런북을 구현 계획에 포함한다.

---

## 1) 로그/트레이싱 기본 원칙

필수:
- 모든 HTTPS Functions에 `requestId`를 생성/전파
- 중요한 상태 변경(승인/환불/정산/웹훅 처리)은 구조화 로그(JSON)로 기록
- PII는 로그에 남기지 않는다(문서 OCR 원문/계좌/주민번호 등 금지)

권장:
- `correlationId`(caseId/sessionId)로 묶을 수 있게 필드 표준화

---

## 2) 메트릭/알림(최소)

결제:
- 웹훅 실패율
- 멱등 충돌/중복 이벤트 카운트

환불:
- 승인 대기 건수(approval pending)
- 환불 실패율

정산:
- 배치 생성 실패/지연
- 미수금 잔액 추이(파트너별/전체)

문서:
- 업로드 실패율
- 검토 지연(SLA)

---

## 3) 런북(Playbook) 최소 세트

1) PG 웹훅 장애
   - 증상: 결제 완료인데 상태 갱신 없음
   - 조치: pgEvent 재처리(멱등), 수동 대사, 로그 조회

2) 환불 분쟁
   - 조치: approval/audit/timeline 근거 링크로 정리, 금액/책임 배분 확인

3) 정산 배치 오류
   - 조치: 배치 입력 이벤트 범위/정책 버전 확인, void 후 재생성

4) 권한 사고
   - 조치: Rules 배포 이력 확인, emulator 테스트 재현, 즉시 롤백

---

## 4) 감사 데이터(필수)

Firestore에 최소한의 감사/타임라인을 남긴다:
- `cases/{caseId}/timeline/{eventId}`: 사용자/파트너/ops가 보는 사실(PII 제거)
- `approvals/{approvalId}`: 승인 패킷 + 결정자 + 사유
- `auditLogs/{logId}`(선택): 운영자/파트너의 민감 액션(PII 조회/환불 집행 등)

> 감사 데이터는 “검색/필터”가 가능해야 하므로, 필요한 필드들은 top-level로 둔다.
