# CS/분쟁/환불 처리 플레이북 — v1

목표: CS(고객지원)와 분쟁/환불은 “문구”가 아니라 **상태 머신 + 동의 로그 + 이벤트**로 운영되어야 분쟁이 줄어든다.  
운영을 AI에게 맡기더라도, AI는 **정책 기반 계산/설명/초안**까지만 하고, 확정은 승인 게이트로 통제한다.

## 0) CS 케이스 분류(티켓 라벨)

CS 접수 시(앱/카톡/전화/이메일) 반드시 아래 중 하나로 라벨링:
- `INFO` : 일반 문의(절차/준비물)
- `DOC` : 문서 업로드/보완
- `QUOTE` : 견적/추가비용/ETA 관련
- `PAYMENT` : 결제/영수증/PG 오류
- `REFUND` : 환불 요청/환불 상태
- `DISPUTE` : 사용자-파트너 분쟁/클레임
- `VISIT` : 방문 예약/노쇼/현장 이슈
- `SECURITY` : 계정/개인정보/문서 유출 의심

> 라벨은 Ops 콘솔에서 큐 라우팅과 SLA 우선순위에 사용.

---

## 1) 공통 처리 원칙(필수)

1. **사실 기반**: 타임라인 이벤트와 문서 버전(증빙)으로만 판단
2. **동의 우선**: 비용/환불/기한 관련은 동의 로그(`quote_consents`)를 확인
3. **1회 메시지 1요청**: 사용자에게 “다음 할 일”은 1개만(이탈/혼란 방지)
4. **외부 메시지는 승인 게이트**: AI/CS 초안 → ops 승인 → 발송
5. **PII 접근은 예외**: 필요 시 사유/만료 포함 승인(기본 마스킹)

---

## 2) 환불 처리 표준 흐름

### 2.1 환불 요청 접수
트리거:
- 사용자 앱에서 환불 요청
- CS가 대리 접수(전화/채팅)

필수 기록:
- 이벤트 `REFUND_REQUESTED`
- 티켓에 `reasonKo` + `reasonCategory`(예: 단순변심/지연/품질불만/중복결제)

### 2.2 환불 가능 범위 판단(정책 기반)
판단 입력:
- 케이스 상태 머신 상 위치
- 문서 진행률(업로드/검토/보완)
- 견적 확정/동의 여부
- 결제 상태(authorized/captured)

출력:
- 환불 가능 금액/불가 사유(정책 설명)
- 부분 환불이면 breakdown(실비/파트너/플랫폼)

> 이 단계는 AI가 잘하는 영역(정책 기반 계산/설명)이며, **확정은 승인**.

### 2.3 승인/집행
- 승인: 이벤트 `REFUND_APPROVED` (ops_approver)
- 집행: PG 환불 성공 시 `REFUND_EXECUTED`

필수:
- `audit_logs`: `APPROVE_REFUND` + 사유
- 부분 환불/정산 차감이 있으면 ledger/settlement에도 반영(대사 큐로 연결)

## 3) 분쟁(사용자-파트너) 처리 표준 흐름

### 3.1 분쟁 유형 분류
- A) 지연/SLA 미준수
- B) 추가 비용 요구/견적 변경
- C) 문서 보완 반복(품질 이슈)
- D) 방문(오프라인) 중 이슈
- E) 커뮤니케이션 문제(응답 없음/불친절)

### 3.2 증빙 수집(필수)
- 타임라인: `domain_events`(메시지/견적/문서/결제)
- 동의 로그: `quote_consents`
- 문서 버전: 업로드/검토 결과(보완요청률 근거)

### 3.3 해결 옵션(정책)
정책 기반으로 아래 중 선택(복수 가능):
- (1) 사용자 안내/정정 메시지(오해 해소)
- (2) 파트너 재검토/재진행(기한 재설정)
- (3) 파트너 변경/재배정(재랭킹)
- (4) 부분 환불/크레딧(플랫폼 이용료 조정 포함 가능)
- (5) 파트너 페널티(정산 차감/노출 제한)

승인 게이트:
- (3)(4)(5)는 L2 이상 권장(ops_approver)

## 4) 메시지 템플릿(초안) — 운영/AI 자동화용

> 실제 문구는 `12-legal/01-terms-privacy-disclosures.md` 고지 문구를 항상 포함.

### 4.1 환불 접수 안내
- 제목: “환불 요청이 접수되었습니다”
- 본문 요지:
  - 현재 사건 상태(예: 문서 검토 전/후)
  - 환불 심사 예상 시간
  - 추가로 필요한 정보 1개(있다면)

### 4.2 환불 거절(또는 제한) 안내
- 제목: “환불 가능 범위 안내”
- 본문 요지:
  - 불가/제한 사유(정책 + 상태 머신 근거)
  - 가능한 대안(재진행/파트너 변경/부분 환불 등)

### 4.3 분쟁 조정 제안
- 제목: “진행 조정안 안내”
- 본문 요지:
  - 사실 요약(이벤트 기반)
  - 제안 옵션 1~2개(선택)
  - 다음 액션 1개(버튼)

---

## 5) CS SLA(응답/해결) — 운영 품질 기준

권장 최소:
- 최초 응답: 1시간 이내(업무시간), 12시간 이내(비업무시간)
- 환불 1차 판단: 24시간 이내
- 분쟁 1차 조정안: 48시간 이내

> 실제 수치는 사업/인력에 따라 조정하되, 시스템은 측정 가능해야 함(관측 문서 참조).

---

## 6) AI에게 위임 가능한 부분 vs 금지

AI 가능:
- 티켓 분류/라벨링
- 정책 기반 환불 범위 계산(초안)
- 타임라인 요약/증빙 링크 수집
- 사용자 메시지 초안 생성(고지 포함)

AI 금지(승인 필요):
- 환불 승인/집행 확정
- 파트너 정지/노출 제한 확정
- PII/OCR 원문 열람
