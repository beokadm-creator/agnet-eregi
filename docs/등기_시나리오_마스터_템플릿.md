# 등기 시나리오 문서 마스터 템플릿

## 문서 목적
- 이 문서는 `AgentRegi`에서 각 등기 케이스를 표준화된 데이터 구조, 질문 퍼널, 산출물 규칙, 파트너 매칭 규칙으로 정의하기 위한 내부 운영용 마스터 템플릿이다.
- 개별 케이스 문서는 반드시 이 템플릿을 복제하여 작성한다.
- 목표는 다음 4가지를 동시에 만족하는 것이다.
  - 유저 퍼널에서 누락 없이 정보 수집
  - 내부 Ops에서 가격/ETA/리스크를 빠르게 산정
  - 파트너에게 필요한 서류와 난이도를 명확히 전달
  - 자동 문서 생성과 보정 대응 규칙을 구조화

## 사용 원칙
- `scenarioKey`는 영문 `snake_case`로 작성한다.
- `등기종류(표준명)`은 유저 노출/파트너 노출/내부 정산에서 동일한 명칭으로 사용한다.
- 문서에는 반드시 `공통 필드`, `케이스 특화 필드`, `예외/리스크 필드`가 모두 있어야 한다.
- 문서에 적는 “필수 산출물”은 실제 접수 전까지 반드시 확보되어야 하는 문서 기준으로 쓴다.
- 법령/규정은 “기본 근거”와 “실무상 추가 확인 필요 포인트”를 구분해서 적는다.
- 관할 등기소, 첨부서면, 인증/번역, 인감/서명, 대표권 방식은 누락 시 보정 가능성이 높으므로 별도 강조한다.

## 1. 범위

### 1-1. 포함
- 이 시나리오에서 처리하는 등기 행위
- 전제되는 사전 검토 항목
- 필수 의사결정 및 결의 절차
- 신청 단계까지 필요한 첨부서면
- 보정 가능성이 높은 리스크와 대응 포인트

### 1-2. 제외
- 별도 라이선스 인허가 변경
- 세무 신고 자체
- 노동/4대보험 후속 절차
- 계약서 검토 전반
- 소송/가처분/분쟁 대응

### 1-3. 유사 케이스 구분 기준
- 같은 UI 퍼널로 묶어도 되는 케이스와 별도 퍼널로 분리해야 하는 케이스를 적는다.
- 아래 기준을 같이 기재한다.
  - 법적 원인행위가 다른지
  - 결의기관이 다른지
  - 필수 입력값이 30% 이상 다른지
  - 산출물 세트가 유의미하게 다른지
  - 파트너 전문영역이 달라지는지

예시:
- `임원 변경`과 `대표이사 변경`은 유사하지만, 대표권 변경 여부와 중임/사임/취임 조합에 따라 산출물과 보정 포인트가 달라 별도 시나리오 또는 depth 2 분기가 필요하다.

## 3. 프로세스

### Step 1. 사전 검토
- 법인 등기부 현재 상태 확인
- 정관 최신본 유무 확인
- 최근 변경 이력과 미등기 사항 존재 여부 확인
- 등기기한 존재 여부 확인
- 이해관계자 신원/대표권/서명 방식 확인
- 관할 등기소 및 관할 특이사항 확인

### Step 2. 원인행위 및 결의
- 어떤 기관이 결의권한을 가지는지 확인
- 이사회/주주총회/사원총회/대표이사 결정 등 결정 구조 확정
- 필요 시 계약, 동의, 승낙, 사임, 인수, 납입 등 원인행위 문서 확보

### Step 3. 서류 작성
- 신청서 초안 작성
- 첨부서면 정합성 확인
- 날짜, 금액, 주소, 성명, 주민등록 정보 일치 여부 확인
- 인감/서명 방식 반영

### Step 4. 납부 및 증빙
- 등록면허세 납부
- 수수료 처리
- 납입/잔고/송금/증명서 등 금융 증빙 확보
- 전자신청 시 파일 형식과 스캔 품질 점검

### Step 5. 등기 신청
- 전자신청 또는 서면신청 방식 선택
- 관할 등기소 재확인
- 첨부서면 누락 여부 최종 점검
- 접수번호 및 예상 처리기간 추적

### Step 6. 보정 대응 및 완료
- 보정명령 수신 여부 확인
- 보정 사유 분류
  - 인적사항 불일치
  - 결의기관 오류
  - 첨부서면 누락
  - 공증/번역/인증 문제
  - 세금/수수료 관련 문제
- 보정 완료 후 등기사항증명서 확인
- 후속 태스크 생성
  - 세무/은행/인허가/계약 변경

  - 예시: `취임, 사임, 중임`
  - 검증: 역할별 조합 가능 여부 검증
  - 저장키: `case.officerChange.actions`
  - 필수 조건: 항상 필수
  - 후속 분기: `사임` 포함 시 사임서/사임일 수집

#### 예시 C. 본점 이전
- 이전 유형
  - 질문: 본점 이전이 같은 관할 내인가요, 다른 관할로 가나요?
  - 입력타입: `single_choice`
  - 예시: `관외이전`
  - 검증: `관내 | 관외`
  - 저장키: `case.headOfficeRelocation.scope`
  - 필수 조건: 항상 필수
  - 후속 분기: `관외`이면 신/구 관할 관련 서류 세트 분기
- 정관상 본점 기재 수준
  - 질문: 정관에 본점이 어떻게 기재되어 있나요?
  - 입력타입: `single_choice`
  - 예시: `서울특별시`
  - 검증: `최소 행정구역 | 상세주소 | 미확인`
  - 저장키: `case.headOfficeRelocation.articlesAddressLevel`
  - 필수 조건: 항상 필수
  - 후속 분기: 상세주소 기재 시 정관 변경 가능성 플래그

### 4-3. 입력값 제약 공통 규칙
- 숫자
  - 금액은 원 단위 정수 저장
  - 수량은 0 초과 정수
  - 퍼센트는 소수 2자리 이내
- 날짜
  - 저장 형식은 `YYYY-MM-DD`
  - 미래/과거 허용 여부를 필드별로 정의
  - 결의일, 취임일, 납입일, 신청예정일의 선후관계 검증
- 텍스트
  - 성명/법인명/주소는 원문 보존 + 정규화본 분리 저장 고려
  - 자유입력 필드는 길이 제한과 금칙문자 정책 정의
- 파일
  - `pdf`, `jpg`, `png` 허용 여부 명시
  - 스캔 해상도, 전체 페이지 누락 여부 확인

### 4-4. 필수 조합 룰
- `대표이사 변경`이면 `취임/사임 여부`, `취임일`, `대표권 방식`, `주민식별정보` 조합이 필수다.
- `자본금 증자`이면 `증자 전/후 자본금`, `발행주식 수`, `1주 금액`, `납입일`, `인수인 정보`가 함께 있어야 한다.
- `본점 관외이전`이면 `구주소`, `신주소`, `결의기관`, `정관 변경 필요 여부`, `관할 등기소 변경`이 함께 있어야 한다.
- `외국인/외국법인` 관련이면 `영문명`, `국적`, `서류 인증 방식`, `번역문`이 함께 있어야 한다.

## 6. 질문 설계
질문은 반드시 `depth 1 → depth 2 → depth 3` 퍼널 구조로 설계한다.

### 6-1. 설계 원칙
- depth 1: 케이스 분류와 가격/난이도에 직접 영향을 주는 질문
- depth 2: 산출물 구성과 법적 절차를 가르는 질문
- depth 3: 문서 작성용 세부값 수집 질문
- 앞 질문의 답으로 뒤 질문이 분기되어야 하며, 불필요한 질문은 숨긴다.
- 질문 하나당 목적을 1개만 둔다.

### 6-2. 질문 정의 포맷
- `id`
- `label`
- `depth`
- `type`
- `required`
- `options`
- `validation`
- `branching`
- `mapsTo`
- `affectsPrice`
- `affectsEta`
- `affectsPartnerMatch`

### 6-3. 질문 예시 포맷
- Q1 (depth 1): 이번에 진행하려는 등기 유형은 무엇인가요? / type: `single_choice` / required: `Y`
  - 옵션: `임원 변경`, `본점 이전`, `자본금 증자`, `상호 변경`, `목적 변경`, `기타`
  - 검증: 1개 선택
  - 다음 분기: 유형별 세부 퍼널 이동
  - mapsTo: `scenarioKey`
- Q2 (depth 1): 법인 형태는 무엇인가요? / type: `single_choice` / required: `Y`
  - 옵션: `주식회사`, `유한회사`, `합자회사`, `기타`
  - 검증: 허용 법인형 목록
  - 다음 분기: 회사 유형별 제출서류 분기
  - mapsTo: `corp.entityType`
- Q3 (depth 1): 언제까지 완료가 필요하신가요? / type: `date` / required: `Y`
  - 검증: 오늘 이후, 법정기한 비교
  - 다음 분기: 긴급 수수료 여부
  - mapsTo: `process.requestedCompletionDate`
- Q4 (depth 2): 최신 정관을 가지고 계신가요? / type: `single_choice` / required: `Y`
  - 옵션: `있음`, `없음`, `모름`
  - 검증: 1개 선택
  - 다음 분기: 정관 확인 필요 플래그
  - mapsTo: `process.hasArticles`
- Q5 (depth 2): 외국인 또는 외국법인이 관련되어 있나요? / type: `single_choice` / required: `Y`
  - 옵션: `예`, `아니오`
  - 검증: 1개 선택
  - 다음 분기: 인증/번역 세부 질문 노출
  - mapsTo: `risk.hasForeignParty`
- Q6 (depth 3): 대표자 성명과 생년월일을 입력해 주세요. / type: `text` / required: `Y`
  - 검증: 포맷/민감정보 마스킹 정책
  - 다음 분기: 없음
  - mapsTo: `stakeholders.representative`

## 7. 가격/ETA/서류 산정 규칙

### 7-1. 기본가
- 기본가는 다음 기준으로 책정한다.
  - 단순 단일 변경
  - 내국인만 관련
  - 정관 최신본 확보
  - 표준 결의 구조
  - 전자신청 가능
  - 보정 이력 없음

### 7-2. 가산/감산 규칙 예시
- 조건 `외국인/외국법인 관련` → `+금액`, `+일수`
- 조건 `관외이전` → `+금액`, `+일수`
- 조건 `공동대표 또는 복수 이해관계자` → `+금액`, `+일수`
- 조건 `복합 사건(증자+임원변경 등)` → `+금액`, `+일수`
- 조건 `정관 부재` → `+금액`, `+일수`
- 조건 `긴급 요청` → `우선순위 가산`
- 조건 `과거 보정/반려 이력 존재` → `리스크 가산`

### 7-3. ETA 규칙
- ETA는 `정보수집 완료일`이 아니라 `필수 입력값과 핵심 서류가 모두 확보된 시점`부터 계산한다.
- ETA 구성 요소를 분리해 적는다.
  - 사전 검토 소요
  - 문서 작성 소요
  - 고객 서명/날인 회수 소요
  - 납부/증빙 준비 소요
  - 등기소 처리 예상 소요
  - 보정 발생 시 추가 소요

### 7-4. 필수서류 목록 규칙
- 기본
  - 법인등기사항증명서
  - 최신 정관
  - 대표자 신원 및 연락 정보
  - 위임장
  - 신청서
- 조건부
  - 의사록
  - 취임승낙서/사임서
  - 주주명부
  - 인감증명서
  - 납입증명/잔고증명
  - 번역문/아포스티유/영사확인
  - 기존 허가증/인가서

## 9. 근거

### 9-1. 기본 근거 법령
- `상업등기법`
  - 상업등기의 목적, 관할 등기소, 등기 접수 및 효력, 등기사무 처리의 기본 틀을 정한다.
  - 관할은 회사 본점 소재지 기준이라는 점을 기본 룰로 본다.
- `상업등기규칙`
  - 신청서 및 첨부서면 작성, 문자 사용, 외국어 첨부서류 번역문, 문서 양식 등 실무 처리 기준을 보완한다.
- `상법`
  - 회사기관, 주주총회, 이사회, 신주발행, 임원 선임/해임, 본점, 상호, 목적 등 원인행위의 실체법상 근거를 제공한다.

### 9-2. 실무상 추가 확인 대상
- 대법원예규
- 선례
- 인터넷등기소 제출 가이드
- 관할 등기소 보정 관행
- 내부 반려/보정 데이터

### 9-3. 관할별 차이 기록 방식
- 항목명:
  - 관할:
  - 이슈:
  - 요구 추가서류:
  - 내부 확인일:
  - 출처:

### 9-4. 실무 팁
- 등기부상 주소, 정관상 주소, 신청서상 주소 표기가 조금이라도 어긋나면 보정 가능성이 올라간다.
- 대표자/임원 성명, 생년월일, 주소는 원문 기준으로 대조하고 자동 정규화값은 별도 저장한다.
- 외국어 서류는 번역문 요구 여부를 보수적으로 판단하고, 인증 방식은 국가별 차이가 있으므로 체크포인트를 별도 둔다.
- 전자신청 가능 여부는 케이스 자체가 아니라 첨부서류의 형식, 서명 방식, 인증 문서 존재 여부에 따라 달라질 수 있다.
- 복합 사건은 한 건으로 묶을지 분리 접수할지 판단 기준을 내부 룰로 축적해야 한다.

- 가격 규칙과 ETA 규칙이 구체적 수치 또는 내부 산식으로 이어질 수 있는가
- 산출물 목록이 실제 접수 기준으로 충분한가
- 외국인/정관특약/공동대표/복합사건 리스크가 검토되었는가
- 법령 링크와 실무 링크가 최신인지 확인했는가

## 15. 조사/작성용 프롬프트(복붙용)

### 15-1. 케이스 문서 작성 프롬프트
```text
너는 ‘대한민국 법인 등기 실무 시나리오’ 조사·정리 전문가다.
목표: 아래 케이스의 등기 업무를 실제로 처리할 수 있을 정도로 “필수 수집 정보 + 프로세스 + 산출물 + 예외/리스크 + 질문 설계(퍼널용)”를 정리해라.

[케이스]
- 등기종류(표준명): {여기에 입력}
- scenarioKey(임시): {예: capital_increase}
- 법인 형태: {주식회사 중심, 필요 시 유한회사/기타 분기}
- 관할: 대한민국 법인 등기(일반). 관할별 차이는 있으면 별도 섹션에 기록.

[출력 형식]
1) 반드시 수집해야 하는 정보를 공통/케이스특화/예외로 나누고, 각 항목마다 (질문 문장 / 입력 타입 single_choice|multi_choice|text|number|date|file / 예시 / 검증 규칙 / 저장키 / 민감정보 여부) 포함.
2) 실무 프로세스를 Step 1~6으로 정리.
3) 필수 산출 문서 목록을 기본/조건부로 나눠 정리.
4) 가격/ETA에 영향을 주는 변수를 목록화(가산/감산 규칙 후보 포함).
5) 퍼널 질문 설계안을 depth 1/2/3으로 나눠 10~25개 정도 제안(불필요 질문은 분기 조건으로 숨김).
6) 마지막에 ‘추가로 확인해야 할 불확실한 포인트’ 목록을 적어라.
7) 가능하면 공식 근거 링크(국가법령정보센터/인터넷등기소)와 실무 관행 포인트를 분리해서 적어라.

[제약]
- 확실하지 않으면 “불확실/추정”이라고 표시하고, 무엇을 확인해야 확정되는지 질문을 써라.
- 숫자/날짜/단위는 명확히 표기해라(원/주/명/일).
- 개인정보/민감정보는 “최소 수집 원칙”으로, 반드시 필요한 경우에만 포함해라.
```

### 15-2. 퍼널 질문 타입 표준(권장)
- `single_choice`: 단일 선택(옵션 고정)
- `multi_choice`: 복수 선택
- `text`: 자유 입력
- `number`: 숫자(원/주/명 등 단위는 label에 포함)
- `date`: 날짜(YYYY-MM-DD)
- `file`: 첨부(스캔본/증빙)

### 15-3. 케이스 문서 작성 작업 순서(권장)
1) 케이스 1개 선택 → 이 템플릿을 복사해 `{scenarioKey}.md`로 생성
2) 0~4번(요약/범위/산출물/프로세스/필수수집)을 먼저 채움
3) 6번(질문 설계)을 depth 1부터 작성 → 부족한 정보는 4번 체크리스트에 되돌려 보강
4) 7번(가격/ETA/서류 규칙) 초안 작성 → 최종은 운영 데이터/파트너 피드백으로 조정
5) 8번(매칭 포인트) 작성 → specialty/tag/tier 기준 확정
