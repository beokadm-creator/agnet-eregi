# 운영 콘솔(Ops Console) 요구사항 — v1

목표: “운영을 AI에게 맡긴다”는 것은 **운영 콘솔이 필요 없다는 의미가 아니라**,  
AI가 수행/추천/자동화를 하더라도 **사람이 통제·감사·롤백**할 수 있는 콘솔이 반드시 필요하다는 뜻입니다.

본 문서는 운영 콘솔이 갖춰야 할 기능을 **큐(Queue) 중심**으로 정의합니다.

---

## 0) 운영 콘솔의 역할(정의)

운영 콘솔은 아래를 보장해야 합니다.

1. **수동검토 안전밸브**: 고리스크/정책 위반/분쟁을 즉시 큐로 라우팅
2. **승인 게이트 실행자**: 메시지 발송/견적 확정/환불 승인 등 “확정” 행위 승인
3. **대사/감사 중심**: 이벤트 기반으로 “왜 이렇게 됐는지” 재현 가능
4. **AI 운영자(오케스트레이터)**: AI가 한 결정을 검증하고, 정책을 적용하고, 실패를 복구

---

## 1) 사용자/권한(Ops RBAC)

### 1.1 Ops 역할(예시)
- `ops_viewer`: 조회만
- `ops_agent`: 케이스/문서/CS 처리를 수행(승인은 제한)
- `ops_approver`: 승인 게이트 수행(환불/메시지/견적)
- `ops_admin`: 정책/권한/설정 변경

권장: DB RLS + 앱 RBAC 병행(상세는 이후 `10-security/01-auth-rbac-rls.md`에서 고정).

---

## 2) 핵심 화면(IA) — Queue First

### 2.1 공통 레이아웃
- 좌측: 큐 목록(배지로 건수/긴급)
- 중앙: 리스트(필터/정렬/검색)
- 우측: 상세 패널(타임라인/문서/AI 제안/액션)

공통 필터:
- 사건팩, 상태, 리스크, SLA 임박, 파트너, 결제 상태, 환불 상태, 최근 이벤트

공통 데이터 소스:
- `cases`, `documents`, `payments/refunds/settlements`, 그리고 **append-only `domain_events`**

---

## 3) 큐 정의(운영이 실제로 “돌아가게” 하는 단위)

### Q1) Manual Review Queue (고리스크/정책 위반)
**진입 조건(예):**
- `RISK_FLAGGED(severity>=high)`
- `CASE_ESCALATED_TO_OPS`
- 문서 위조/신원확인 이슈(신호 기반)
- 파트너 정책 위반(광고/추천/허위정보)

**리스트 컬럼(최소):**
- caseId, 사건팩, 현재 상태, 리스크 레벨/사유, 파트너, 마지막 이벤트 시각, SLA 임박

**상세에서 가능한 액션:**
- (승인) `manual_review -> in_progress` 재개
- (재배정) 파트너 변경/해제 + 재랭킹 트리거
- (정지) 케이스 동결(필요 시)
- (정책 적용) 광고/추천 제외 플래그 적용

**AI 지원(운영에 맡길 부분):**
- 리스크 요약(근거 이벤트/문서 ref)
- “권장 액션” 1~3개 제안(확정은 사람)

---

### Q2) Approval Gate Queue (승인 대기)
AI/파트너/시스템이 “확정 전 단계”에서 멈추는 큐.

대상:
- 메시지 발송(보완 요청, 안내)
- 견적 확정(추가 비용/추가 서류 포함)
- 환불 승인

상세에서 반드시 보여야 하는 것:
- **무엇을 승인하는지**(payload diff)
- **근거(sourceRefs)**: 사건팩 규칙/문서/정책
- 영향 범위: 사용자에게 어떤 문구가 나가며, 금액/기한이 어떻게 바뀌는지

액션:
- 승인/반려(반려 사유 템플릿)
- 수정 후 승인(텍스트/금액 일부 수정)
- “정책 업데이트 필요” 플래그(반복 발생 시)

---

### Q3) SLA Breach / At-Risk Queue
목적: SLA가 깨지기 전에 AI/운영이 개입.

진입 조건:
- `SLA_BREACH_DETECTED` 발생
- 또는 “임박(예: 2시간 이내)” 선제 경고

액션:
- 사용자/파트너 알림 발송(승인 게이트 적용 가능)
- 자동 에스컬레이션(Manual review)
- 재배정 제안/실행

---

### Q4) Refund / Dispute Queue (환불/분쟁/클레임)
진입 조건:
- `REFUND_REQUESTED`
- 클레임 태그(사용자 CS 접수)

상세에서 필요:
- 사건 상태(상태 머신 기준)
- 견적/결제/문서 진행률(환불 정책 적용 근거)
- 사용자 동의 로그(`quote_consents`)

액션:
- 환불 승인/거절(사유 기록)
- 부분 환불 계산(정책 기반)
- 파트너 패널티/정산 차감(정책 기반, 감사 로그 필수)

---

### Q5) Settlement Reconciliation Queue (대사/정산 이상)
진입 조건:
- 결제/환불 이벤트와 원장/정산이 불일치
- 정산 배치 실패
- 광고 과금 이슈(노출/클릭 중복, 부정 클릭 의심)

액션:
- 재계산/재정산(배치 재실행)
- 이중 지급/누락 지급 탐지/수정
- 부정 트래픽 필터 적용(광고)

---

## 4) 타임라인(이벤트 뷰) 요구사항

케이스 상세에는 반드시 “이벤트 타임라인”이 있어야 합니다.
- 표준: `02-engine/events/case_event.schema.json`
- 표시: eventType, occurredAt, actor, producer, 핵심 data 요약
- 기능: 필터(문서/결제/리스크), 원문 JSON 보기, 연관 이벤트(correlationId) 묶기

---

## 5) 운영 콘솔에서 AI를 “맡기는” 방식(필수 가드레일)

AI 자동화 원칙:
1. **AI는 큐를 만든다(분류/요약/추천)**  
2. **사람/정책이 확정한다(승인 게이트)**  
3. 모든 결정은 **이벤트/감사로그**로 남는다

운영 콘솔은 AI 관련해서 아래 기능이 필요:
- AI 제안 카드(근거/불확실성 포함)
- “승인/반려/수정” UI + diff view
- AI 기능 플래그(엔진별 on/off)
- 모델/프롬프트/정책 버전 표시(재현/분쟁 대비)

---

## 6) 감사로그(Audit) 요구사항

모든 운영 액션은 `audit_logs`에 남는다.
- VIEW_PII, APPROVE_*, OVERRIDE_*, REASSIGN_PARTNER, FREEZE_CASE 등
- 사유(reason) 필수(티켓/메모)
- 원문 payload 스냅샷 또는 hash 저장

---

## 7) “없으면 운영이 멈추는” 최소 기능 체크리스트
- [ ] Manual review 큐 + 케이스 상세 + 액션
- [ ] 승인 게이트 큐(메시지/견적/환불)
- [ ] SLA 임박/브리치 큐
- [ ] 환불/분쟁 큐
- [ ] 이벤트 타임라인(JSON 원문/필터)
- [ ] 감사로그(액션/사유/추적)
