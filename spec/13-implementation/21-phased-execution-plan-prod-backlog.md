# 프로덕트 개발 미완 기능 백로그 — Phase별 순차 실행 계획(v1)

목표: 현재까지 구현된 “연결된 코어 플로우” 위에서, 남은 미완 기능을 **의존성 기반으로 단계화**하고, 각 단계별 **완료 기준(DoD)** 을 명확히 하여 누락/데드코드를 방지한다.

현 상태(요약)
- Phase 0 기반: 공통 HTTP/에러, Auth/RBAC, 타임라인, 멱등성, CI(빌드+rules test) ✅
- Phase 1~3 일부: 케이스/문서/견적/승인/결제/환불 핵심 경로가 “연결된 형태”로 동작 ✅
- 단, 다수는 **스텁/간소화** 상태(특히 PG/정산/운영/테스트)

---

## Phase 0 — 기반/보안/테스트(프로덕 착수 게이트)

### P0-1 Rules 최종화(권한 사고 방지)
범위:
- sessions/events 소유자 제한
- documents collectionGroup 조회의 안전성 검증
- cases/documents/quotes/payments/refunds/approvals/idempotencyKeys 전 경로에 대해 “클라 직접 write 금지”와 “참여자 read” 일관성 유지

DoD:
- rules_test가 아래 시나리오를 모두 커버하며 CI에서 통과
- 의도치 않은 collectionGroup 탐색 쿼리로 타 케이스 데이터가 새지 않음

### P0-2 Storage Rules 강화 + (가능하면) 테스트
범위:
- slotId별 MIME/크기 정책(예: 신분증/등본/위임장)
- 업로드/삭제 권한(삭제는 ops만)

DoD:
- 대표 슬롯 3개 이상 정책 확정 및 적용
- 업로드 실패/권한 실패가 UX로 확인 가능

### P0-3 Dev endpoint 운영 차단(사고 방지)
범위:
- `/v1/dev/*`는 에뮬레이터에서만 동작

DoD:
- production에서 호출 시 항상 403

---

## Phase 1 — “실사용 화면 단위” UI/흐름 완성(User/Partner/Ops)

핵심: 지금은 “데모 UI” 수준이므로, 실제 운영에 필요한 화면 단위를 완성한다.

### P1-1 User App
- 케이스 상세(타임라인/문서 상태/견적 상태/결제 상태)
- 보완 요청 시 UX(어떤 슬롯을 어떻게 보완해야 하는지)
- 결제 완료/실패 UX

DoD:
- 유저가 “케이스 시작→문서 제출→견적 수락→결제”를 앱에서 끝낼 수 있음

### P1-2 Partner Console
- 케이스 상세: 문서 목록/버전/검토/메모(PII 금지), 견적 작성/확정 요청
- 문서 검토 큐 고도화(필터/정렬/슬롯별)

DoD:
- 파트너가 “케이스 수락→문서 검토→견적 제시”를 콘솔에서 수행 가능

### P1-3 Ops Console
- approvals 큐/결정/감사
- 환불 집행(승인 후 실행) + 재시도 안전

DoD:
- 승인/환불 관련 운영 업무가 콘솔에서 끝남

---

## Phase 2 — 문서/검토/슬롯 정책의 실체화(업무 품질)

### P2-1 슬롯 카탈로그 확정
- casePackId별 요구 문서 슬롯 목록/필수/옵션
- 슬롯별 검토 기준(최소 체크리스트)

DoD:
- “무엇을 제출해야 케이스가 진행되는지”가 시스템적으로 결정됨(규칙/코드/UX)

### P2-2 검토 루프 자동화 정교화
- needs_fix 사유 구조화(코드/카테고리)
- 재제출 시 자동 상태 복귀 규칙 정교화

DoD:
- 문서 보완 루프가 케이스 상태 머신과 충돌 없이 동작

---

## Phase 3 — PG 실연동(가장 위험/가장 중요)

결제는 “스텁”에서 “실거래 안전”으로 전환해야 한다.

### P3-1 결제 상태 머신 확정
- created → authorized → captured/failed/cancelled
- 각 상태 전이의 단일 실행 경로(웹훅/재처리)

### P3-2 웹훅 서명 검증 + 멱등 + 재처리 큐
- 서명 검증 필수
- pgEventId 기반 멱등
- 실패 이벤트 재처리(ops action 또는 자동 재시도)

### P3-3 환불(부분/전체) 정책 확정
- 승인게이트 연동
- 웹훅/상태 대사

DoD:
- 중복 웹훅/네트워크 재시도로 결제가 “두 번 처리”되지 않음
- 장애 상황에서 재처리로 정상 회복 가능

---

## Phase 4 — 정산/미수금(수익/회계/운영 핵심)

### P4-1 정산 배치 생성(Scheduler)
- 기간별 정산 생성
- 지급 대상/금액 산출

### P4-2 지급/상계/미수금
- 상계 한도/최소 지급 가드레일
- 미수금 open/closed lifecycle

### P4-3 리포트/감사
- 정산 리포트(파트너/ops)
- 감사 로그 연결

DoD:
- 월말 정산이 시스템으로 생성/검증/집행 가능
- 분쟁/감사에 필요한 근거가 타임라인/로그로 재현 가능

---

## 진행 순서(권장)
1) Phase 0(P0-1~3) — “권한/보안/CI”를 완전히 잠그고 시작  
2) Phase 1 — 화면 단위로 실사용 완성  
3) Phase 2 — 문서 슬롯 정책을 확정해 품질/일관성 확보  
4) Phase 3 — PG 실연동(고위험)  
5) Phase 4 — 정산/미수금(사업 운영)
